📄 Project: Cyber-Superego (赛博超我)
—— 基于 LangGraph 的边缘-云端混合架构 (Edge-Cloud Hybrid) 个人智能体系统

1. 项目背景与愿景 (Project Context)
本项目旨在开发一个高度模块化、保护隐私且运行成本极低的通用型个人生活监督 Agent。
为了解决传统 VLM（视觉大模型）全天候云端调用的高昂 API 成本和隐私泄露风险，本系统首创**“双脑协同（Dual-Brain Synergy）”机制：
将“沉重的视觉感知”下放到本地边缘端计算，将“复杂的逻辑决策”上传至云端 LLM 计算，最后回传本地接管软硬件权限进行“物理/数字级惩罚执行”**。

2. 核心架构设计 (Core Architecture: Edge-Cloud Hybrid)
系统底层由 LangGraph 状态机驱动，整体工作流表现为一个不断循环的“感知 -> 决策 -> 执行”的有向无环图 (DAG)。

整个链路分为三大核心节点 (Nodes)：

👁️ 节点 A：本地边缘感知层 (Local Edge Perception)
职责： 充当系统的“眼睛”，负责 24 小时低成本提取物理世界的信息，并将其转化为纯文本。

工作流：

通过 OpenCV 定时调用 Mac 本地摄像头截取当前环境帧，并在人上面画绿框，根据moondream2输出的行为，再把行为也放到绿框旁边。

将图像数据输入到本地部署的轻量级视觉模型（如 Ollama 驱动的 Moondream2 ）。

本地模型根据提示词进行推理（例如：“描述画面中人物的行为，判断其是否处于健康/学习状态”）。

输出： 输出一段纯文本的场景描述（如：“宿主正躺在床上玩手机，未在书桌前学习”）。该文本将写入全局 State。

🧠 节点 B：云端中枢决策层 (Cloud Brain Decision)
职责： 充当系统的“大脑”，负责结合用户设定的“生活宪法”进行高级逻辑推理，并下达惩罚判决。

工作流：

读取全局 State 中由本地感知层生成的“纯文本场景描述”。（注意：此阶段绝不向云端发送任何图像数据，确保绝对隐私与极低 Token 成本）。

结合系统提示词（System Prompt，设定为一个极其严厉、毒舌的教导主任），调用云端高性能 LLM API（如 GPT-4o 或 Claude 3.5 Sonnet）。

LLM 判断当前宿主行为是否违规。如果不违规，结束当前循环；如果违规，LLM 必须从绑定的工具库中选择一个最合适的惩罚工具。

输出： 返回带有 tool_calls 的标准化消息对象。

🦾 节点 C：本地物理/数字执行层 (Local Tool Execution)
职责： 充当系统的“手”，接收云端的判决并对宿主进行实质性打击。

工作流：

拦截云端 LLM 发出的 tool_calls。

使用 LangGraph 的原生工具节点机制，在 Mac 本地环境中调用对应的 Python 函数。

工具库（Tools Library）是高度热插拔的，包含但不限于：

OpenClaw/GUI 自动化工具： 强行接管鼠标键盘，执行如“打开微信发送社死表白”、“强制关闭游戏窗口”等操作。

TTS 语音通道： 播放 LLM 生成的嘲讽语音。

打开网页工具，能够打开学习网页或者其他整活的

输出： 执行结果反馈回全局 State，完成闭环。

3. LangGraph 状态机设计 (State & Routing Design)

3.1 状态定义 (State Schema)
图的全局状态需要维护以下核心信息：

messages: 标准的对话消息列表，用于云端 LLM 维持上下文和工具调用记录。

current_vision_text: 本地 Moondream2 模型输出的当前环境纯文本描述。

healthy：是否健康

timestamp: 触发当前判定的系统时间（用于 LLM 判断是否在特定的作息时间内）。

3.2 路由逻辑 (Conditional Edges)
起点 (START) 必然指向 感知节点 (Node A)。

感知节点 (Node A) 提取完文本后，根据条件判断，如果视觉模型显示不健康行为指向 决策节点 (Node B)，并上传具体行为描述。

决策节点 (Node B) 之后引入条件边 (Conditional Edge)：

如果 LLM 决定不需要惩罚（无 tool_calls），图流转至 结束 (END)。

如果 LLM 决定惩罚并返回了 tool_calls，图流转至 执行节点 (Node C)。

执行节点 (Node C) 运行完工具后，返回 决策节点 (Node B)（由 LLM 确认惩罚是否完毕，随后结束）。

4. 给 AI 编程助手 (Cursor) 的开发指导原则
为了保证代码的可读性、可维护性以及后续“整活功能”的无限扩展，请严格遵循以下原则编写代码：

高度模块化 (Modularity)：

将工具库单独放在 tools.py 中，每个工具必须带有清晰的 @tool 装饰器和详尽的 Docstring（因为 LLM 依赖 Docstring 来选择工具）。

将 LangGraph 的节点定义和图的构建逻辑放在 graph.py 中。

主运行逻辑放在 main.py 中。

Mock First (测试驱动)：

在第一版代码生成时，为了便于快速验证跑通图的流转逻辑，请先为“本地视觉感知 (Moondream2)”和“本地物理执行 (OpenClaw 发微信)”编写 Mock（模拟）函数。

例如：感知节点直接返回硬编码的字符串 "时间 08:05，宿主正躺在床上刷手机"；执行节点只在终端打印 "模拟执行：鼠标已接管，发送微信社死文本"。

日志友好 (Verbose Logging)：

在状态流转的每一个节点，必须在终端打印出极其清晰、带有颜色区分的日志（如 [边缘感知]... -> [云端决策]... -> [本地执行]...），以完美呈现“边缘-云端混合流转”的极客视觉效果。
